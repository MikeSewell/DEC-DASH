**Change 1 — Add enrollments table (insert between `clients` and `sessions` table definitions, after line 162):**

```typescript
enrollments: defineTable({
  clientId: v.id("clients"),
  programId: v.id("programs"),
  status: v.union(
    v.literal("pending"),
    v.literal("active"),
    v.literal("on_hold"),
    v.literal("completed"),
    v.literal("withdrawn")
  ),
  enrollmentDate: v.number(),
  exitDate: v.optional(v.number()),
  exitReason: v.optional(v.string()),
  notes: v.optional(v.string()),
  createdBy: v.id("users"),
  updatedAt: v.number(),
})
  .index("by_clientId", ["clientId"])
  .index("by_programId", ["programId"])
  .index("by_status", ["status"]),
```

Key decisions:
- 5-state status lifecycle: pending / active / on_hold / completed / withdrawn (locked decision)
- `createdBy` is `v.id("users")` (required, not optional) — enrollments always created by an authenticated user; migration scripts use internalMutation with admin user ID
- `updatedAt` is `v.number()` (required) — per CONTEXT.md decision for status tracking
- `exitReason` is `v.optional(v.string())` — for completed/withdrawn outcome reporting
- `enrollmentDate` is `v.number()` (Unix ms timestamp) — consistent with project standard for dates

**Change 2 — Add 5 optional fields to clients table (append after `createdAt: v.number(),` and before the closing `}`):**

```typescript
  // v2.0 demographic fields (Phase 16)
  gender: v.optional(v.string()),
  referralSource: v.optional(v.string()),
  dateOfBirth: v.optional(v.string()),    // ISO "YYYY-MM-DD"
  phone: v.optional(v.string()),
  email: v.optional(v.string()),
```

Key decisions:
- `gender` and `referralSource` use `v.optional(v.string())` (not union type) — per discretion finding: intake form data is free-form with inconsistent terminology; string is flexible for Phase 18 migration; analytics can normalize at query time
- `dateOfBirth` stored as ISO string "YYYY-MM-DD" (locked decision) — simpler migration from intake form string data
- All 5 fields wrapped in `v.optional()` — additive change, existing documents remain valid

CRITICAL: Preserve the existing `.index("by_programId", ["programId"])` chain on the clients table. Do not remove or modify it.

Replace the existing index chain with the expanded chain (preserving existing `by_clientId`):

- `by_enrollmentId` index: fast enrollment-scoped queries (Phase 17+)
- `by_sessionDate` index: date-range queries for Phase 19 analytics rewrite (critical performance)
