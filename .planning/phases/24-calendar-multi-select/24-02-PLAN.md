  <action>
Rewrite `src/components/admin/GoogleCalendarConfig.tsx` to replace the manual calendar ID input with a multi-select calendar picker. The new component flow:

**State management:**
- `availableCalendars`: `Array<{ id: string; summary: string }>` — fetched from the API
- `selectedIds`: `Set<string>` — calendar IDs the admin has checked
- `fetching`: boolean — loading state for the fetch button
- `saving`: boolean — loading state for save
- `syncing`: boolean — loading state for sync now
- `message`: string — success/error feedback

**On mount / initial load:**
- Initialize `selectedIds` from `config.calendars` (map `calendarId` values into a Set) via useEffect, same as current component does.
- Do NOT auto-fetch available calendars on mount — user clicks "Fetch Calendars" to trigger.

**"Fetch Calendars" button:**
- Calls `listCalendars()` from the `useListCalendars` hook.
- On success, sets `availableCalendars` state. If the result is empty, show a warning message: "No calendars found. Make sure calendars are shared with the service account."
- On error, show error message.
- Show `fetching` spinner on the button during the call.

**Calendar list UI (shown after fetch):**
- Render a bordered rounded-lg container with divide-y rows (matching the existing calendar list style).
- Each row has:
  - A checkbox input (`<input type="checkbox">`) on the left
  - The calendar summary (bold, text-sm)
  - The calendar ID in small monospace text below the summary (text-xs text-muted font-mono)
- Checkbox is checked if the calendar's `id` is in `selectedIds`.
- Clicking a checkbox toggles the calendar in/out of `selectedIds`.
- If a calendar from `config.calendars` is NOT in the fetched `availableCalendars` list (stale/removed), show it at the bottom in a separate "Previously configured (not found)" section with a warning style. This handles the edge case where a calendar was shared then un-shared.

**"Save" button:**
- Maps `selectedIds` to the `calendars` array format expected by `saveConfig`: for each selected ID, find the matching entry in `availableCalendars` to get the `summary` as `displayName`. If the ID came from a previously-configured calendar not in the fetched list, use the existing `displayName` from config.
- Calls `saveConfig({ calendars })`.
- On success, optionally triggers a sync and shows success message.
- Disabled when `selectedIds` is empty AND `availableCalendars` has been fetched (allow saving empty to clear config).

**"Sync Now" button:**
- Same behavior as current component — calls `triggerSync()`.
- Only shown when config is already saved with at least one calendar.

**Layout structure (top to bottom):**
1. Status header (green/yellow dot + "Configured"/"Not Configured" badge + last sync time) — keep existing
2. Info text: "Fetch available calendars from your Google service account, then select which ones to sync."
3. "Fetch Calendars" button (primary style, with refresh icon)
4. Calendar checkbox list (only visible after fetch)
5. Message area (success/error)
6. Action buttons row: "Save Selection" (primary) + "Sync Now" (secondary, conditional)

**Styling:**
- Match existing Card, Button, Spinner imports.
- Use `cn()` for conditional classes.
- Checkbox styling: use standard HTML checkbox with `accent-color` set to primary or use `className="w-4 h-4 rounded border-border text-primary focus:ring-primary"`.
- Keep the existing pattern of bordered list items for the calendar rows.
- Use the warm design system: rounded-lg borders, text-sm body, text-muted for secondary text.

**Important: Remove the old manual input fields** — no more `newCalendarId`/`newDisplayName` text inputs or "Add Calendar" form. The entire calendar configuration is now done through the fetch-and-select flow.
